# **Экономическая спецификация: «Бизнес с Контуром»**

**Версия:** 1.0  
**Назначение:** задать все числовые параметры, формулы и зависимости для реализации игрового цикла.  
**Формат для разработчика:** конфигурационные файлы (JSON/CSV) с возможностью правки без перекомпиляции.

## **1. Базовые параметры бизнесов**


| Параметр | Магазин | Кофейня | Салон |
| --- | --- | --- | --- |
| Стартовый капитал | [50 000 ₽] | [40 000 ₽] | [60 000 ₽] |
| База клиентов/день (до модов) | [80] | [100] | [40] |
| Средний чек базовый | [300 ₽] | [180 ₽] | [800 ₽] |
| Пропускная способность базовая (чел/день) | [60] | [70] | [30] |
| Налог (процент от дохода) | 6% | 6% | 6% |
| Склад: есть/нет | да | да | нет |
| Срок годности (дней) | [10] | [7] | — |
| Убыток от просрочки (% от стоимости) | [80%] | [80%] | — |
| Снижение убытка с Маркетом (% от обычного убытка) | [20%] (т.е. убыток 64% от стоимости) | аналогично | — |

**Сезонные модификаторы (по месяцам):**

- Магазин: январь [0%], февраль [0%], … июль [+5%], август [+5%], …

- Кофейня: декабрь–февраль [–15%], июнь–август [+22%], остальные [0%]

- Салон: март–май [+12%] (свадебный сезон), остальные [0%]

## **2. Клиенты и пропускная способность**

### **2.1 Формула расчёта количества клиентов в день**

```
Клиенты_база = значение из таблицы бизнеса
Сезонный_мод = коэффициент из сезонной таблицы (например, 0.85 для –15%)
Рекламный_мод = сумма эффектов активных рекламных кампаний (например, +0.15 для +15%)
Репутационный_мод = функция от репутации:
    репутация 0–20 → 0.5
    21–40 → 0.7
    41–60 → 0.9
    61–80 → 1.0
    81–100 → 1.1
Событийный_мод = временный модификатор от событий (например, +0.2)
```

```
Клиенты_всего = round( Клиенты_база * (1 + Сезонный_мод + Рекламный_мод + Событийный_мод) * Репутационный_мод )
```

### **2.2 Пропускная способность**

```
Пропускная_база = значение из таблицы бизнеса
Маркет_бонус = если Маркет активен, то +0.2 (т.е. +20%)
Персонал_бонус = если лояльность >80, то +0.1; если лояльность <30, то –0.15
Улучшения_бонус = сумма бонусов от купленных улучшений (например, расширение зала +0.4)
```

```
Пропускная_итог = round( Пропускная_база * (1 + Маркет_бонус + Персонал_бонус + Улучшения_бонус) )
```

### **2.3 Обслуженные и упущенные**

```
Обслужено = min( Клиенты_всего , Пропускная_итог )
Упущено = Клиенты_всего – Обслужено
```

### **2.4 Влияние упущенных на репутацию**

```
Репутация_изменение_за_день = –0.2 * Упущено
```

Репутация не может упасть ниже 0 и подняться выше 100.

## **3. Доходы и средний чек**

### **3.1 Базовый средний чек (из таблицы бизнеса)**

### **3.2 Модификаторы чека**

```
Чек_мод = 
    +0.15 если активен Маркет
    + сумма от активных улучшений (например, «премиум-категории»)
    – от рекламных кампаний (например, промо-акция –0.2)
    – от событий
```

```
Средний_чек_итог = round( Средний_чек_база * (1 + Чек_мод) )
```

### **3.3 Доход за день**

```
Доход = Обслужено * Средний_чек_итог
```

## **4. Расходы (за день и за месяц)**

### **4.1 Ежедневные расходы (списываются каждый день)**

```
Налог = Доход * 0.06
Подписки_день = сумма_стоимости_активных_подписок_в_месяц / 30
Закупки_день = стоимость товара, списанного со склада по FIFO (только для бизнесов со складом; для салона = 0)
```

### **4.2 Закупки (только для склада)**

- **Таймер закупки:** каждые [3] дня автоматически предлагается закупить новую партию. Размер партии = [48] единиц (пример). Стоимость единицы = [5 ₽] (пример).

- Если остаток на складе становится меньше прогноза на 2 дня (прогноз = средний расход за последние 3 дня), то панель закупки появляется внепланово.

- **Пустой склад:** если в начале дня запас = 0, кнопка «Следующий день» блокируется, пока игрок не сделает закупку.

### **4.3 FIFO и просрочка**

- Каждая партия хранит дату поступления и срок годности.

- Ежедневно при проверке склада: любая партия, у которой возраст ≥ срок годности, списывается с убытком.

```
Убыток_за_партию = количество_просроченных_единиц * цена_закупки_единицы * (0.8 или 0.64 с Маркетом)
```

- Списание происходит до расчёта закупок за день.

### **4.4 Ежемесячные расходы (каждые 30 дней)**

```
Аренда = [15 000 ₽] (зависит от бизнеса и улучшений)
Зарплаты = [база 20 000 ₽ + 5 000 ₽ за каждого доп. сотрудника]
Полные подписки = сумма_стоимости_активных_подписок_в_месяц (без деления на 30)
```

Эти расходы списываются одной суммой в начале 30-го дня.

## **5. Репутация (шкала 0–100)**

- **Стартовая репутация:** [50]

- **Изменения:** от упущенных клиентов (см. п. 2.4), от событий, от синергий (например, Маркет+ОФД дают +2 в день).

- **Влияние репутации на клиентов** — через репутационный мод (п. 2.1).

- Если репутация = 0 в течение 3 дней подряд → поражение.

## **6. Лояльность персонала (шкала 0–100)**

- **Стартовая лояльность:** [60]

- **Падение:** если загрузка (Обслужено / Пропускная_итог) > 0.9 в течение 3 дней подряд → –[10] лояльности в день начиная с 4-го дня.

- **Рост:** выдача премии — 5% от выручки текущего дня → +[15] лояльности.

- **Влияние:** лояльность >80 → +10% к пропускной; лояльность <30 → –15% к пропускной.

- **Отображение:** точные цифры только с сервисом Эльба. Без Эльбы — иконка с примерным уровнем (низкий/средний/высокий).

## **7. Сервисы Контура (подписки)**

Каждый сервис имеет:

- **Стоимость в месяц** (см. таблицу в GDD)

- **Эффекты** (модификаторы формул)

- **Синергии** (дополнительные эффекты при совместном подключении)

Все эффекты и синергии должны быть реализованы как флаги, влияющие на соответствующие коэффициенты в формулах выше.

## **8. Рекламные кампании**

Каждая кампания описывается:


| Название | Длительность (дней) | Стоимость | Эффект на клиентов | Эффект на чек | Доступность |
| --- | --- | --- | --- | --- | --- |
| Промо-акция | 10 | 3000 ₽ | +25% | –20% | все бизнесы |
| Счастливые часы | 10 | 2500 ₽ | +20% | –15% | только кофейня |
| Яндекс.Директ | 30 | 8000 ₽ | +15% | 0% | все |
| … и т.д. (12 штук) |  |  |  |  |  |

Ограничение: нельзя запустить две одинаковые кампании одновременно. После окончания кампании она становится доступна снова.

## **9. Улучшения (покупаются за деньги, разовые)**

Пример структуры:


| Улучшение | Стоимость | Эффект | Дополнительные условия |
| --- | --- | --- | --- |
| Расширение зала | [50 000 ₽] | +40% пропускной | –30% клиентов на 5 дней (стройка), +15 000 ₽/мес аренды |
| Наём администратора | [10 000 ₽] | +20% пропускной | +5 000 ₽/мес зарплаты |
| Премиум-категории | [30 000 ₽] | +15% к чеку | требует Маркет |

## **10. События (база данных)**

Для разработчика нужна структура записи события:

- `id`, `название`

- `триггер`: день, диапазон дней, репутация, наличие сервиса, случайный шанс (например, 5% в день)

- `текст описания`

- до 3 вариантов выбора, у каждого:

  - текст кнопки

  - изменения ресурсов (деньги, репутация, лояльность, склад)

  - требования к сервису (если вариант доступен только при активном сервисе)

  - флаг `is_contour_option` для визуального выделения

Пример строки (CSV):

```
TAX01;Налоговая проверка;день>10, реп<50;Пришла налоговая...;Разбираться самому;-30000, -5;Нанять юриста;-18000, -2;Экстерн;-5000,0;Экстерн;true
```

## **11. Прогрессия и достижения**

### **11.1 Уровни (1–10)**

Опыт за:

- +1 опыт за каждый прожитый день

- +X опыта за заработанные деньги (например, +1 за каждые 10 000 ₽ чистой прибыли)

- +Y опыта за повышение репутации выше 80

- +Z опыта за каждый подключённый сервис

Таблица уровней (пример):


| Уровень | Опыт | Что открывается |
| --- | --- | --- |
| 1 | 0 | базовые улучшения |
| 2 | 100 | реклама «Флаеры» |
| … | … | … |
| 10 | 1000 | финальное достижение |

### **11.2 Достижения**

Структура:

- `id`, `название`, `условие` (логическое выражение), `награда` (тип скидки, сервис, длительность)

- Пример: `ACH_NO_SPOIL;Не пропало ни грамма;10 дней без просрочки;discount;Маркет;0.5;15` (скидка 50% на Маркет на 15 дней)

## **12. Параметры поражения и победы**

**Поражение, если:**

- Баланс денег становится отрицательным после списания ежемесячных расходов.

- Репутация = 0 в течение 3 дней подряд.

**Победа, если выполнены ВСЕ:**

- Чистая прибыль за день ≥ [100 000 ₽]

- Накоплено на счету ≥ [1 000 000 ₽]

- Подключены все 7 сервисов

- Уровень ≥ 10

- Заработано ≥ 7 достижений

## **13. Хранение параметров (для разработчика)**

Все числовые параметры должны выноситься в конфигурационные файлы (JSON/CSV), чтобы балансёр мог менять их без перекомпиляции. Минимальный набор файлов:

- `business_config.json` — параметры трёх типов бизнеса

- `seasonal_mods.json` — сезонные коэффициенты

- `services_config.json` — сервисы, их стоимость, эффекты, синергии

- `ads_config.json` — рекламные кампании

- `upgrades_config.json` — улучшения

- `events.csv` — библиотека событий

- `achievements.json` — достижения и награды

- `level_table.json` — пороги опыта и открываемые фичи
